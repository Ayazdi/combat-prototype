# Draft of Blades — Element System Refactor

## Context

The current prototype lives in `draft_combat_prototype.jsx` (single file, React). It implements a tile-drafting combat system with three tile types: Attack (A), Defence (D), Empty (E). The player drafts 4 tiles across 4 rounds into a sequence; consecutive same-type tiles combo for damage/block multipliers. A 6-enemy gauntlet exists with passive abilities (charged_strike, empty_plus, no_first_defence, reroll_lock, double_discard). Shield persists between fights (capped at 40).

This refactor **replaces the core combat mechanic** with an element-based system while keeping the visual frame, gauntlet structure, log system, stat bar layout, and tuning-constants pattern.

## Tiles

### Element tiles
- **Steel (S)** — physical, reliable. No status effects. Best raw damage scaling.
- **Fire (F)** — damage-over-time. Stacks burn.
- **Ice (I)** — control. Stacks freeze (amplifies physical damage).

### Utility tile
- **Empty (·)** — takes a hand slot, contributes nothing in any action. Generated by RNG only.

### Draft weights
| Tile | Weight |
|---|---|
| Steel | 30 |
| Fire | 25 |
| Ice | 25 |
| Empty | 20 |

Anti-degenerate rule: if all 6 draft slots are the same tile, reroll one slot to a different element (keep current `rollRow` style).

## Hand Allocation (new core mechanic)

After 5 rounds of drafting, the player has 5 tiles in hand. Before resolution, the player allocates each tile to one of three actions:

- **Attack** — deal damage to enemy
- **Shield** — gain block (caps at `maxShield`)
- **Boost** — bank charges of that element

Free split: any combination is valid (5/0/0, 3/1/1, 0/2/3, 0/0/5). Tiles cannot be left unassigned. Allocation completes when all 5 tiles are placed; only then does Resolve become available.

### UI suggestion
Keep the 5-slot committed row visible. Below it, render three labeled buckets (Attack / Shield / Boost) each with a drop area or assignment buttons. Each tile can be reassigned freely until Resolve is clicked. Each bucket shows a **live preview**:

- Attack bucket: total damage, burn stacks to apply, freeze stacks to apply
- Shield bucket: total block, contact effects, contact stacks
- Boost bucket: charges to bank per element

Recommended interaction: click a tile in hand, then click a bucket. Or three buttons under each hand tile (A / S / B). Pick whichever feels cleanest in React; drag-and-drop is not required.

## Combos (within a single action bucket)

Within one action bucket, tiles of the same element combo. Order within the bucket is irrelevant — count the elements.

| Count | Multiplier |
|---|---|
| 1 | 1.0× |
| 2 | 1.3× |
| 3 | 1.6× |
| 4 | 2.0× |
| 5 | 2.8× |

Combos are computed **per element per action**. Steel and Fire in the same Attack bucket combo separately, then sum. Empty contributes nothing and does not break combos (since order doesn't matter inside a bucket).

---

## Element Mechanics

Damage and block formulas use `base × count × combo`, where `combo` is the multiplier for that element's count in the bucket.

### Steel
| Action | Effect |
|---|---|
| Attack | `12 × count × combo` damage. Affected by enemy freeze stacks (see Freeze). |
| Shield | `10 × count × combo` block |
| Boost | Banks Steel charges |

### Fire
| Action | Effect |
|---|---|
| Attack | `6 × count × combo` damage + `count` burn stacks applied to enemy |
| Shield | `4 × count × combo` block + `count` burn-on-contact stacks (applied to attacker if shield is hit) |
| Boost | Banks Fire charges |

**Burn**: deals 4 HP/turn at end of the victim's turn for 2 turns. Stacks add. Cap 10 stacks per target. Each new application **refreshes duration** to 2 turns.

### Ice
| Action | Effect |
|---|---|
| Attack | `5 × count × combo` damage + `count` freeze stacks applied to enemy |
| Shield | `8 × count × combo` block + `count` freeze-on-contact stacks |
| Boost | Banks Ice charges |

**Freeze**: each stack adds +15% **physical (Steel)** damage taken, multiplicative. Decays 1 stack per turn (end of victim's turn). Cap 5 stacks per target. Does NOT amplify Fire or Ice base damage — Steel only.

## Mixing Rules (the heart of the system)

When tiles of different elements share a bucket:

### Steel + Fire (or Steel + Ice)
**Additive.** Each element resolves independently with its own combo math, then sum.

Example — Attack [3 Steel, 2 Fire]:
- Steel: 12 × 3 × 1.6 = 57.6 → **58 dmg**
- Fire: 6 × 2 × 1.3 = 15.6 → **16 dmg + 2 burn stacks**
- Total: **74 dmg + 2 burn stacks**

### Fire + Ice (opposing) — Cancellation, 1:1 ratio
Pairwise cancellation between Fire and Ice tiles **in the same bucket**.

Each Fire-Ice pair cancels:
- Both elements lose their **status effect** (the cancelled tiles produce no burn, no freeze)
- The cancelled tiles still deal **raw element damage at 1.0×** (no combo)
- Surviving singletons of either element keep their normal effect AND combo math, computed on the surviving count only

Example — Attack [3 Fire, 2 Ice]:
- 2 pairs cancel: 2 fire + 2 ice
  - Cancelled damage: (2 × 6) + (2 × 5) = 12 + 10 = **22 dmg**, no status
- 1 fire surviving: 6 × 1 × 1.0 = **6 dmg + 1 burn stack**
- Total: **28 dmg + 1 burn stack**

Example — Attack [4 Fire, 1 Ice]:
- 1 pair cancels: 1 fire + 1 ice → 6 + 5 = **11 dmg, no status**
- 3 fire surviving: 6 × 3 × 1.6 = 28.8 → **29 dmg + 3 burn stacks**
- Total: **40 dmg + 3 burn stacks**

### Mixed across different buckets
No interaction. Fire in Attack and Ice in Shield is fine — they don't cancel.

### Steel never cancels
Steel has no opposing element. Steel + Fire and Steel + Ice are always purely additive.

## Boost

### Banking
Tiles in the Boost bucket bank charges of their element. Track separately:

```js
charges = { steel: [...timestamps], fire: [...timestamps], ice: [...timestamps] }
```

Each charge carries the turn it was banked (for expiry).

**Diminishing returns within a single boost action**: the 1st, 2nd, and 3rd tile of an element banked in the same action contribute 1.0 charge each. The 4th and 5th contribute 0.5 charges each (round to nearest charge per element at the end of the action, floor on .5).

**Total charge cap**: 5 per element. Excess discarded (logged).

### Spending
Charges are consumed automatically by the **next** action that uses that element:

- An attack with Fire tiles spends Fire charges first. Each charge = 1 virtual extra Fire tile **for the purpose of damage, status, AND combo math**, without occupying a hand slot.
- A shield with Steel tiles spends Steel charges. Same logic.
- Charges always spend (always beneficial); no opt-out.

Example: 2 banked Fire charges + this turn's Attack has 2 Fire tiles → resolves as if 4 Fire tiles for damage, burn stacks, AND combo curve.
- 6 × 4 × 2.0 = **48 dmg + 4 burn stacks**

A bucket with 0 tiles of an element does NOT spend that element's charges.

### Expiration
Charges expire 3 turns after being banked, oldest first. Track turn-of-banking per charge.

### Reset between fights
Charges clear at end of each fight. Do not carry between gauntlet stages.

## Enemy Turn Structure

Each enemy turn now telegraphs:
1. **Attack**: `damage` + `element` (Steel / Fire / Ice). Status follows from element type.
2. **Shield** (optional): `block` + `element`.

These are pre-set per enemy attack pattern (scripted), not drafted. Enemies do not boost.

### Resolution: player attack vs enemy shield

When the player's Attack bucket resolves and the enemy has shield up:

1. Compute total player attack damage and statuses (after combos and Fire-Ice cancellation among player's own tiles).
2. **Element interaction with enemy shield (1:1 cancellation)**:
   - If enemy shield is **Fire** and player attack contains **Ice tiles** (after step 1's internal cancellations): cancel 1:1. Each surviving ice tile cancels one fire-shield burn-on-contact stack. Each cancelled fire-shield stack also reduces the shield element's contact damage by its base.
   - If enemy shield is **Ice** and player attack contains **Fire tiles**: same, opposite direction. Each fire tile cancels one ice-shield freeze-on-contact stack.
   - Steel: no element interaction, just damage absorption.
3. Subtract shield from raw damage. Damage past shield deals full hit.
4. Apply attack's status effects to enemy (modulo cancellations from step 2).
5. Apply shield's contact effect to player (burn stacks if fire shield, freeze stacks if ice shield), modulo cancellations.

### Resolution: enemy attack vs player shield

Symmetrical. Player's Shield bucket may have Fire/Ice tiles whose contact stacks apply to the enemy on hit, modulo cancellation by the enemy attack's opposing element.

### Detailed cancellation example

**Setup**: Player attacks with [3 Fire, 2 Steel] (already resolved internally: 16+1 from steel? let me redo):
- Steel: 12×2×1.3 = 31 dmg (no status)
- Fire: 6×3×1.6 = 29 dmg + 3 burn stacks
- Total: 60 dmg + 3 burn stacks

**Enemy shield**: Ice 24 (3 ice tiles, 8×3×1.0)

**Resolution**:
- Player has 3 fire tiles, enemy has 3 ice shield tiles → 3 pairs cancel
- Player's burn stacks: 0 applied (3 cancelled)
- Enemy shield's freeze contact: 0 applied to player (3 cancelled)
- Damage: 60 - 24 = **36 dmg to enemy**, no status either side

### Order of operations per turn

```
1. Player drafts hand (5 rounds × 1 pick from row of 6)
2. Player allocates 5 tiles to Attack/Shield/Boost
3. Player clicks Resolve. Resolution begins:
   a. Player Boost: bank charges (with diminishing returns and cap)
   b. Player Shield: spend Steel/Fire/Ice charges, compute block, set incoming
      shield buffer with element tag(s) and contact effects
   c. Player Attack: spend Steel/Fire/Ice charges, compute damage and statuses
      (including internal Fire-Ice cancellation)
      → Interact with enemy shield (1:1 cancellation between attack element
        and shield element)
      → Apply remaining damage to enemy HP (Steel damage amplified by enemy
        freeze stacks)
      → Apply remaining attack statuses to enemy
      → Apply shield contact effects to player (modulo cancellation)
   d. If enemy HP ≤ 0: victory, stop
   e. Enemy Attack: compute damage and statuses
      → Interact with player shield (same cancellation logic)
      → Apply remaining damage to player HP (Steel damage amplified by player
        freeze stacks)
      → Apply remaining attack statuses to player
      → Apply player shield contact effects to enemy (modulo cancellation)
   f. If player HP ≤ 0: defeat, stop
4. End-of-turn ticks (in this order):
   a. Burn ticks: for each side with burn stacks, deal stacks × 4 dmg, then
      decrement duration. If duration = 0, clear stacks.
   b. Freeze decay: each side with freeze loses 1 stack.
   c. Charge expiration: charges older than 3 turns are removed.
   d. Player mana regen.
5. Increment turn counter, start next draft phase.
```

### Freeze damage interaction

Freeze stacks on a target multiply incoming **Steel damage only** at the moment of damage calculation:

```
finalSteelDamage = baseSteelDamage × (1 + freezeStacks × 0.15)
```

Fire and Ice base damage are not affected by freeze.

## Open Decisions (defaults assumed; flag to override)

1. **Cancellation damage rule**: cancelled fire/ice tiles in the same bucket deal their base element damage at 1.0× (no combo, no status). _Alternative: cancelled tiles deal 0 damage._
2. **Burn refresh**: new application refreshes duration to 2 turns. _Alternative: stack durations independently._
3. **Charge persistence between fights**: cleared at end of fight. _Alternative: persist with shield._
4. **Wild tiles**: omitted from v1. _Add later if drafting feels too punishing._
5. **Empty tiles**: pure dead weight (just take a slot). _Allocation already separates buckets, so a "breakpoint" mechanic is unnecessary._
6. **Enemy passive abilities** (charged_strike, empty_plus, etc.): removed. The element system provides enough variety. _Add back as a future polish pass if needed._
7. **Steel charges from boost**: spent only when Steel tiles are in the relevant action. A pure-Fire attack does not consume Steel charges.
8. **Order of bucket resolution**: Boost → Shield → Attack within player's turn. (Boost first so its charges don't apply this turn; Shield before Attack so shield is up when player attack is calculated against the enemy. Enemy attack then resolves against the player's just-built shield.)
